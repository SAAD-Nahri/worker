# Source Review Checklist

## Purpose

This checklist defines how sources and individual source items should be reviewed during the manual-first pilot.

The goal is to keep intake disciplined. A source should not enter the pipeline just because it is available. An item should not enter the pipeline just because it exists.

## Two Levels Of Review

There are two review layers:

1. source-level review,
2. item-level review.

Source-level review decides whether a domain belongs in the registry.
Item-level review decides whether a specific article or post is worth transforming.

## Part 1: Source-Level Review

Use this checklist before adding a new source to the active seed set.

### A. Scope Fit

Ask:

1. does this source regularly publish food-related content,
2. does it produce food facts, ingredient explainers, kitchen myths, food origins, or curiosity-friendly food topics,
3. is it aligned with the first niche rather than being too general or too health-heavy.

### B. Intake Efficiency

Ask:

1. does the source have usable RSS or at least stable article structure,
2. can it be fetched without unreasonable friction,
3. will maintaining this source be reasonable for a solo operator.

### C. Transformation Value

Ask:

1. are the source items good raw material for short explanatory blog content,
2. do they support template-based transformation,
3. can they produce Facebook-friendly curiosity packaging.

### D. Quality Risk

Ask:

1. is the content too sensational,
2. is it too dependent on health claims,
3. is it too noisy, thin, or structurally messy,
4. is it so recipe-heavy that filtering becomes inefficient.

### E. Diversification Value

Ask:

1. does this source add something different from the existing seed set,
2. does it reduce single-source dependence,
3. does it improve range across source families.

## Source-Level Decision Guide

### Accept

Accept the source if:

1. it strongly fits the niche,
2. it is operationally realistic,
3. it has repeatable transformation value,
4. it improves the seed mix.

### Watchlist

Put the source on a watchlist if:

1. the content fit is good but intake is annoying,
2. the brand is attractive but the content mix is noisy,
3. it is worth revisiting after the first pilot.

### Reject

Reject the source if:

1. the content is too health-claim heavy,
2. it is mostly recipes with little explanatory value,
3. intake is too fragile for v1,
4. it adds no meaningful value to the source mix.

## Optional Source Scoring

For manual review, use a 1-to-5 score in each category:

1. scope fit,
2. intake efficiency,
3. transformation value,
4. quality safety,
5. diversification value.

Practical recommendation:

1. `22+` = strong seed candidate,
2. `18-21` = watchlist candidate,
3. below `18` = reject for now.

### Operational Rule For Week One

For the first live pilot:

1. a source should only remain active if it scores `22+` and has no hard red flag,
2. a hard red flag includes severe recipe-only bias, strong health-claim dependence, or operational friction that clearly breaks the solo-operator model,
3. a source may remain selective instead of primary if quality is mixed but diversification value is still useful.

## Part 2: Item-Level Review

Use this checklist when reviewing specific source items from approved domains.

### A. Topic Fit

Ask:

1. is this clearly about a food fact, food origin, ingredient explanation, kitchen myth, or food curiosity angle,
2. can this become one of the approved v1 blog template types.

### B. Blog Conversion Fit

Ask:

1. can this be turned into a short explanatory blog post,
2. is there a clean core angle,
3. is there enough material to support structure without filler.

### C. Facebook Packaging Fit

Ask:

1. does this item have a clear curiosity hook,
2. could it plausibly drive a Facebook click,
3. is the angle understandable in one or two short lines.

### D. Quality And Risk

Ask:

1. is the item too close to a recent draft or published piece,
2. is it too thin,
3. does it rely on claims we do not want to amplify,
4. would transforming it create derivative-risk or low-value output.

### E. Queue Value

Ask:

1. is this worth spending a publish slot on,
2. does it help diversify the current batch,
3. does it fit the current pilot goals.

## Item-Level Decision Guide

### Advance

Advance the item if:

1. it fits the niche,
2. it fits a template,
3. it has a clear hook,
4. it is not a weak duplicate.

### Operational Rule For Week One

For the first live pilot, advance an item only if:

1. topic fit is clearly positive,
2. blog conversion fit is clearly positive,
3. Facebook packaging fit is at least acceptable,
4. there is no hard reject in quality and risk,
5. the item feels strong enough to compete for a real publish slot.

### Hold

Hold the item if:

1. it is promising but unclear,
2. it needs comparison against other queued items,
3. it may be usable later.

### Reject

Reject the item if:

1. it is too weak,
2. it is too repetitive,
3. it is too risky,
4. it is not worth the effort to transform.

## Recommended Manual Intake Workflow

For the pilot, the intake workflow should be:

1. review source feed,
2. shortlist candidate items,
3. run item-level checklist,
4. mark each item as advance, hold, or reject,
5. only move approved items into the drafting queue.

This is intentionally simple and professional. The goal is fast judgment with clear standards.

## Definition Of Done

This checklist is working when:

1. sources are not added casually,
2. items are not transformed casually,
3. the pilot queue contains intentionally chosen candidates rather than random feed noise.
