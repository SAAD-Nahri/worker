## Purpose

These specs answer the questions that usually cause weak starts:

1. what exact records exist,
2. what states content moves through,
3. what counts as duplicate,
4. what templates must exist first,
5. what a draft record must contain,
6. what quality gates should block or flag drafts,
7. what derivative-risk means in practice,
8. what AI is and is not allowed to do,
9. what category and tag policy controls classification,
10. what the formatting pipeline should do stage by stage,
11. how content-fit and anchor quality should be evaluated before trusting a draft,
12. how weak-fit items should be rerouted, held, or rejected,
13. how draft review should work,
14. how operators should read draft health from runtime state,
15. what gets recorded after publishing,
16. what always requires human approval,
17. what a WordPress-ready publish payload must contain,
18. what Facebook-ready package records and template families must look like,
19. what queue states mean during distribution,
20. how blog and Facebook outputs stay linked before deeper tracking exists,
21. how social package approval should be recorded,
22. how local WordPress publish-state updates should progress before full transport exists,
23. how local Facebook publish-state updates should preserve publish identifiers, queue transitions, and final mapping state before live transport exists,
24. how manual versus auto scheduling should remain explicit, conservative, and review-safe in v1,
25. how the first real WordPress transport adapter should stay draft-only, dry-run-safe, and append-only,
26. how the first real Facebook transport adapter should stay Page-focused, dry-run-safe, and append-only,
27. how the operator should read distribution-chain health, collisions, and broken-state alerts from runtime records,
28. how the operator should validate WordPress and Facebook transport configs before live use,
29. how the operator should plan schedule-ready and collision-heavy publish chains,
30. what the first normalized publish chain snapshot should look like,
31. how the append-only runtime records should be normalized into a reportable publish history layer,
32. how source, draft, blog, social, Facebook, queue, and mapping records should be joined without guessing,
33. how selected variants should be preserved without pretending v1 is an experiment system,
34. which logs matter for audit and debugging versus pure noise,
35. what first reporting views should exist before deeper analytics starts,
36. which Phase 4 audit records are justified and how they should stay low-noise,
37. when normalized chain snapshots should remain on demand versus when persistence would actually be justified,
38. how local transport credentials and config files should be handled safely during live operator activation,
39. how later provider-backed AI can improve text quality without becoming a foundation dependency,
40. how later media and visual assets should be handled safely and reviewably,
41. how the operator should see the current Phase 4.5 activation blockers in one explicit readiness report instead of guessing from several disconnected views,
42. how a real remote WordPress post state should be reconciled back into the append-only local chain after a manual admin publish or schedule step,
43. how the finished system should actually run in production without depending on the developer workstation,
44. how recurring jobs should be scheduled and deployed without turning the repo into infrastructure theater,
45. how backup and recovery should work for a solo-operator runtime,
46. how future approval automation can remain narrow, explainable, and reversible instead of becoming reckless autopublish,
47. how human approval should fit into the operator UI without turning the project into a giant CMS,
48. how the internal operator API can expose review-safe workflows without creating a second workflow engine,
49. how the WordPress admin plugin can act as a thin review shell without becoming the source of truth,
50. how the operator API should actually be run locally and in production once WordPress admin depends on it.

## Rule

If implementation decisions conflict with these specs, update the specs first or document why they are wrong. Do not let code silently redefine the architecture.
